# crew-pwa/ — Crew-side PWA infrastructure (Stage 8)

Offline support, service-worker registration, IndexedDB caching, and the
sync layer for the crew PWA. All user-facing UI for the crew landing,
install banner, offline indicator, and sync-status display lives in
`MobileCrewView.jsx` and will be rebuilt in Stage 9 — these modules are
headless plumbing.

## File map

| File | Role |
| --- | --- |
| `indexedDbSchema.js` | Opens `surveyos_crew_v1` DB (`idb` wrapper), declares every object store + index, exports shared sync-status constants, and owns the version-upgrade path. |
| `syncManager.js` | Pull / push / merge lifecycle. Subscribes to `online/offline` events. Drains the `pending_writes` queue when online. Applies the "field fields win for crew, metadata fields win for PM" merge policy. |
| `swRegistration.js` | Wraps `virtual:pwa-register` from `vite-plugin-pwa`. Idempotent, prod-only. |
| `installPrompt.js` | Captures `beforeinstallprompt`, exposes `triggerInstall`, persists the "never show again" dismissal state to `localStorage`. |
| `useCrewSync.js` | Single React hook Stage 9 should consume. Unifies sync + install state with action callbacks. |

## How Stage 9 should consume this

```jsx
import useCrewSync from '../crew-pwa/useCrewSync';
import { initializeSync } from '../crew-pwa/syncManager';

export default function MobileCrewView({ supabase, profile }) {
    useEffect(() => {
        if (profile?.id && profile?.firm_id) {
            initializeSync({
                supabase,
                userId: profile.id,
                firmId: profile.firm_id,
            });
        }
    }, [supabase, profile?.id, profile?.firm_id]);

    const {
        isOnline,
        pendingWriteCount,
        lastSyncAt,
        showInstallBanner,
        installApp,
        dismissInstallBanner,
        manualSync,
    } = useCrewSync();

    // …render crew UI, install banner, sync badge, etc.
}
```

**Rule of thumb**: Stage 9 UI should not import from `syncManager.js` or
`installPrompt.js` directly — the hook is the contract.

## Sync policy

Pull scope (`pullAssignmentData`):

- `stakeout_assignments` where `status ∈ {draft, sent, in_progress, submitted}`
  and `assignment_date` is within `CACHE_WINDOW_DAYS` (7) of today.
- All `stakeout_assignment_points` / `stakeout_design_points` /
  `stakeout_qc_points` rows for those assignments.
- RLS is the source of truth for per-crew visibility — we just ask for
  what we can see.

Push policy (`pushPendingWrites`) — one item at a time, oldest first:

- **`stakeout_qc_points` insert** → always accepted; UUID is client-generated
  so collision is impossible.
- **`stakeout_qc_points` update** → filtered to crew-owned columns
  (`field_fit_reason`, `field_fit_note`, `built_on_status`). Other columns
  are stripped — the RLS column-protection trigger from migration 12
  would reject them anyway.
- **`stakeout_assignments` update** → crew never owns columns on this
  table in Phase 1. Queued updates are dropped with a debug log and the
  PM's remote state is preserved.
- **`stakeout_assignment_points`** → PM-only. Queued writes are dropped.

Conflict resolution is handled client-side (column filter on push) rather
than server-side (409 detection), because the RLS trigger guarantees the
wire-level rejection would happen anyway. This avoids a roundtrip.

## Cache lifecycle

- `last_sync_at` is stored in the `meta` store.
- On `initializeSync`, if `last_sync_at` is missing or older than
  `CACHE_WINDOW_DAYS` and we're online, a full pull kicks off.
- Stage 9's manual-sync button calls `manualSync()` which pulls then
  pushes.
- Switching users (different `userId` in `initializeSync`) clears the
  entire local cache via `clearAllCrewData()`.

## Service worker

- Generated by `vite-plugin-pwa` (Workbox) at build time.
- Only registered in production (`import.meta.env.PROD`). Dev hot-reload
  stays service-worker-free.
- Runtime caching: `NetworkFirst` with 5-second timeout for Supabase REST
  endpoints, cached under `supabase-api-v1` (max 200 entries, 7-day
  maxAge). Static app shell is precached.
- Push notifications are **deferred** to Stage 9. No `push` listener
  exists yet.

## What Stage 8 explicitly does **not** ship

- Crew UI (Stage 9 owns `MobileCrewView.jsx`).
- Push notifications.
- Background Sync API integration (the `pending_writes` queue is drained
  via the foreground `online` event; Stage 10 can upgrade to
  `SyncManager.register('queue-flush')` if needed).
- Real app icons — `scripts/generate-pwa-icons.cjs` emits solid-teal
  placeholders. Stage 13 polish replaces them.

## Legacy notes

Two older service-worker files exist at `public/sw.js` and
`public/sw-vault.js`. The former is a pass-through from the Pillar 3
prototype; the latter was an early background-sync experiment registered
from `src/lib/syncEngine.js`. Both will be superseded by the
Workbox-generated worker. Stage 10 will clean them up once the new path
is battle-tested in a trial deployment.
